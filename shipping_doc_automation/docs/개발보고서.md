## 1. 프로젝트 개요

### 1.1 배경

HD현대일렉트릭 배전전략영업과에서 월 200건에 달하는 선적서류를 담당자 2인이 수작업으로 작성 중이며,
개인 경험에 의존하는 PL(Packing List) 작성의 자동화가 필요하다.

### 1.2 목표

CI(Commercial Invoice) 엑셀 파일을 입력받아 PL(Packing List)을 자동 생성하는
Python CLI 프로토타입을 개발한다.

### 1.3 접근 방식

**규칙 기반 알고리즘 + ML 보정 (하이브리드)**

```
CI 엑셀 파싱 → 제품 분류 → 포장 전략 선택 → 적재 최적화 → 중량 산정 → PL 엑셀 출력
```

### 1.4 검증 데이터

5개 실제 CIPL 샘플 엑셀 파일을 기반으로 개발 및 검증을 수행하였다.

### 4.3 CI 파서 (`src/parser/ci_parser.py`)

**.xls / .xlsx 양형식** 지원 (xlrd / openpyxl):

- **시트 자동 탐지**: `"CI"`, `"1818-1 CI"` 등 패턴 매칭
- **동적 헤더 탐색**: `"Description of Goods"` 텍스트 검색
- **데이터 시작점 탐지**: 카테고리 헤더 또는 주문번호 헤더 기반

- **합적 주문 감지**: `S-TOTAL` / `G-TOTAL` 패턴 + `^\d{8}(-\d+)?$` 주문번호 정규식
- **카테고리 헤더 구분**: Bold + C열만 값 + F열(수량) 비어있음

### 4.4 제품 분류기 (`src/classifier/product_classifier.py`)

**3단계 분류 전략** (우선순위 순):

```
0단계: SPARE 헤더 검사 → "SPARE PART FOR..." 헤더면 즉시 SPARE
1단계: 카테고리 헤더 기반 → CI 문서 구조에서 가장 신뢰도 높음
2단계: 모델번호 접두사 기반 → 헤더 없는 경우 폴백
3단계: Description 텍스트 → 최종 폴백
```

**설계 결정**: 카테고리 헤더를 모델번호보다 우선하도록 설계하였다.
이유: 동일 모델(예: UCB)이 문맥에 따라 MCCB 또는 ACB_LARGE로 분류될 수 있으며,
CI 문서에서 카테고리 헤더가 해당 섹션의 제품 유형을 명시적으로 정의하기 때문이다.

**ACB 프레임 세분화**:
- A-FRAME (800A 이하) → 케이스당 최대 6대
- B-FRAME (1600A 이하) → 케이스당 최대 4대
- C-FRAME (2000A 이상) → 케이스당 최대 2대

### 4.5 포장 전략 엔진 (`src/packing/`)

| 패커 | 대상 카테고리 | 알고리즘 | 규칙 |
|------|-------------|----------|------|
| `vcb_packer` | VCB | 1:1 매핑 | 1대 1케이스, 모델별 규격 룩업 |
| `acb_packer` (standard) | ACB_HGS | First-Fit Decreasing | 프레임별 그룹, 최대 4~6대/case |
| `acb_packer` (large) | ACB_LARGE | 1:1 매핑 | HGN/UAN 개별 대형 케이스 |
| `mixed_pallet_packer` | MCCB/MC/Relay/SPARE | Greedy Bin Packing | 400kg 제한, 카테고리 우선순위 정렬 |

**혼합 팔레트 적재 알고리즘 상세**:

```
1. 전체 품목을 카테고리별 우선순위 정렬 (MCCB→MC→Relay→SPARE)
2. 동일 카테고리 내에서 무게 내림차순 정렬
3. Greedy Bin Packing:
   a. 기존 팔레트에 넣을 수 있으면 추가 (잔여 중량 확인)
   b. 넣을 수 없으면 새 팔레트 생성
   c. 팔레트당 순중량 400kg 제한
4. 팔레트 → PackedCase 변환 (같은 모델 그룹핑)
```

**합적 주문 처리**:
- 주문별 독립 패킹 → 순차 케이스 번호 부여
- PL 출력 시 A열에 주문번호, E열에 `(S-TOTAL)` 표기

### 4.6 PL 엑셀 생성기 (`src/generator/pl_generator.py`)

샘플 PL과 동일한 레이아웃으로 엑셀 파일을 생성한다.

**레이아웃 구조**:
```
[PL 타이틀]
[Sheet 정보]
[송하인 정보]
[수하인/Notify]
[Port/Carrier]
[컬럼 헤더: Marks | Description | Qty | N.W | G.W | Measurement]

  ...
[CASE NO : 2 of N]

```

**주요 서식 규칙**:
- 카테고리명 복수형 자동 변환 (`"CIRCUIT BREAKER"` → `"CIRCUIT BREAKERS"`)
- 케이스 헤더 행에 N.W/G.W/규격, 카테고리 행에 CBM 배치
- 합적 주문 시 주문번호/S-TOTAL 표기
- TOTAL 행에 합계 (수량, 순중량, 총중량, CBM)

### 4.7 검증 모듈 (`src/validation/pl_validator.py`)

5개 자동 검증 항목:

| 항목 | 검증 내용 | 판정 |
|------|----------|------|
| 수량 일치 | CI 총수량 = PL 총수량 | Error |

## 5. ML 중량 예측 모듈

### 5.1 개요

미등록 제품의 단위중량을 예측하기 위한 보조 ML 모듈이다.
주 중량 결정은 `product_db.json` 룩업 + 규칙 기반 추정이며,
ML은 두 방법 모두 실패할 때의 최종 폴백으로 동작한다.

### 5.2 알고리즘: GradientBoostingRegressor

**핵심 원리**: 약한 학습기(결정 트리)를 순차적으로 학습시켜, 각 트리가 이전 트리들의 잔차(오차)를 보정하는 앙상블 기법이다.

```
F₀(x) = mean(y)                          # 초기: 전체 평균
F₁(x) = F₀(x) + η × tree₁(residual₁)    # 1차 보정
F₂(x) = F₁(x) + η × tree₂(residual₂)    # 2차 보정
...
Fₙ(x) = Fₙ₋₁(x) + η × treeₙ(residualₙ) # n차 보정
```

여기서 `residualₜ = y - Fₜ₋₁(x)`는 손실함수의 음의 기울기(negative gradient)이며,
이것이 "Gradient" Boosting이라는 이름의 유래이다.

### 5.3 피처 설계

모델번호에서 5개 수치 특성을 추출한다:

| 피처 | 설명 | 예시 (HGM630) |
|------|------|---------------|
| `family_id` | 제품 계열 ID | 6 (HGM) |
| `log_current` | log(전류정격+1) | log(631) ≈ 6.45 |
| `frame_type` | 프레임 타입 인코딩 | 0 |
| `frame_size` | 프레임 크기 인코딩 | 4 (630AF) |
| `log_base_weight` | log(계열 기본중량) | log(4) ≈ 1.39 |

### 5.4 과적합 분석 및 해결

초기 구현에서 **심각한 과적합**이 발견되어 수정하였다.

#### 원인 분석

| 요인 | 상세 |
|------|------|
| 데이터 부족 | 60개 샘플, 피처 7개 (비율 8.6:1, 권장 50:1 이상) |
| 타겟 편향 | 0.14kg~200kg, **1,429배 범위** |
| 과도한 복잡도 | 100 트리 × depth 4 → 수천 리프노드가 60개에 맞춤 |
| 피처 중복 | `current_rating` = `model_numeric` (동일값) |

#### 진단 지표

| 지표 | 수정 전 | 수정 후 |
|------|---------|---------|
| Train R² | 0.817 | 0.466 |
| **CV R² (5-fold)** | **-16,647** | **-4.28** |
| Train MAE | 4.9 kg | 12.3 kg |
| **CV MAE** | **23.1 kg** | **12.0 kg** |
| **Train-CV MAE 갭** | **18.2 kg** | **0.3 kg** |

#### 적용한 과적합 방지 전략

| 전략 | 변경 내용 | 효과 |
|------|----------|------|
| **Log 변환** | 타겟 y에 log 적용 | 1,429배 범위 → 7.27 범위 |
| **모델 단순화** | n_estimators: 100→30, max_depth: 4→2 | 모델 용량 대폭 축소 |
| **정규화** | min_samples_leaf=3, min_samples_split=5 | 리프 과적합 방지 |
| **Stochastic GB** | subsample=0.8 | 20% 데이터 누락으로 분산 감소 |
| **피처 정리** | 7→5개 (중복 제거, log 정규화) | 차원 축소 |

#### 하이퍼파라미터 비교

```python
# 수정 전 (과적합)
GradientBoostingRegressor(
    n_estimators=100, max_depth=4, learning_rate=0.1
)

# 수정 후 (정규화)
GradientBoostingRegressor(
    n_estimators=30,        # 트리 수 축소
    max_depth=2,            # 트리 깊이 제한
    learning_rate=0.1,
    min_samples_leaf=3,     # 리프 최소 샘플
    min_samples_split=5,    # 분할 최소 샘플
    subsample=0.8,          # 확률적 부스팅
)
```

### 5.7 학습 데이터 확장 전략

NNLS(Non-Negative Least Squares)를 사용하여 기존 PL에서 단위중량을 역산한다:

```
케이스 순중량 = Σ(모델별 수량 × 단위중량)

→ 연립방정식 A·x = b 구성
  A[i,j] = 케이스 i에서 모델 j의 수량
  b[i]   = 케이스 i의 총 순중량
  x[j]   = 모델 j의 단위중량 (미지수)

→ scipy.optimize.nnls로 비음수 해 도출
```
